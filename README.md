# CampusGuide AI

An AI assistant that answers student administrative questions (lost ID cards,
certificates, hostel admission, scholarships, etc.) using a structured
knowledge base of 50 campus services, Google Gemini for natural-language
answers, and a live analytics dashboard.

## Architecture

- **Backend:** FastAPI (`main.py`) — REST API for chat, service lookup,
  feedback, analytics, and authentication.
- **Data layer:** `store.py` — uses **PostgreSQL** (via Supabase) when
  `DATABASE_URL` is set; automatically falls back to an in-memory store if
  it isn't, so the app can never crash on startup because of a database
  issue. Tables: `services`, `chat_history`, `feedback`, `users`, `sessions`.
- **Retrieval:** hybrid semantic + lexical search (`find_best_match` in `main.py`).
  Every service is embedded once at startup using Google's `text-embedding-004`
  model; each user query is embedded at request time and matched by cosine
  similarity, blended with keyword overlap (65% semantic / 35% lexical) for
  robustness on short or typo-prone queries. This is genuine vector retrieval —
  not a local ChromaDB index, but Gemini-hosted embeddings, chosen so the app
  has no heavy ML dependencies (no `torch`/`sentence-transformers`) and stays
  fast and reliable to deploy on a free Render instance. If no `GEMINI_API_KEY`
  is set, or an embedding call fails, retrieval automatically falls back to
  pure keyword matching — check `GET /health` to see which mode is active
  (`hybrid-semantic` vs `keyword-only`).
- **LLM:** Google Gemini (`gemini-2.0-flash`) generates the final answer
  from the matched service's structured data. If no `GEMINI_API_KEY` is
  set, or the call fails, the app falls back to a template-generated
  answer built from the same structured fields — so the demo never breaks
  on stage.
- **Auth:** email/password signup and login. Passwords are hashed with
  PBKDF2-SHA256 (200,000 iterations, random salt per user) — no plaintext
  password is ever stored. Sessions are opaque random tokens stored server-side
  with a 7-day expiry, sent by the frontend as `Authorization: Bearer <token>`.
- **Frontend:** a single self-contained `static/index.html` (no build step,
  no external CDN dependency) — chat UI with structured result cards,
  a searchable service directory, and a dashboard with KPI cards and
  bar charts.

## Local setup

```bash
pip install -r requirements.txt
export GEMINI_API_KEY=your_key_here        # optional — falls back to templates if unset
export DATABASE_URL=postgresql://...        # optional — falls back to in-memory if unset
uvicorn main:app --reload
```

Visit `http://localhost:8000`.

## Setting up PostgreSQL (Supabase, free tier)

1. Create a project at [supabase.com](https://supabase.com).
2. Go to **Project Settings → Database → Connection string** and copy the
   **URI** (use the "Connection pooling" string on port 6543 for
   serverless-friendly connections).
3. Set it as the `DATABASE_URL` environment variable (locally, or on
   Render — see below).
4. That's it — on first startup the app automatically creates all tables
   (`users`, `sessions`, `services`, `chat_history`, `feedback`) and seeds
   the `services` table from `services.json`. No manual migration needed.

## Deployment (Render)

1. Push this repo to GitHub.
2. On [Render](https://render.com): **New → Web Service**, connect the repo.
3. Build command: `pip install -r requirements.txt`
4. Start command: `uvicorn main:app --host 0.0.0.0 --port $PORT`
5. Environment variables:
   - `GEMINI_API_KEY` — your Gemini API key
   - `DATABASE_URL` — your Supabase connection string
6. Deploy. Render gives you a public URL — that's your live demo link.

## API reference (quick)

| Method | Endpoint        | Description                              |
|--------|-----------------|-------------------------------------------|
| POST   | `/auth/signup`  | Create account → returns session token    |
| POST   | `/auth/login`   | Log in → returns session token            |
| GET    | `/auth/me`      | Current user (requires `Authorization`)   |
| POST   | `/auth/logout`  | Invalidate session token                  |
| POST   | `/chat`         | Ask a question, get an AI answer + service card |
| GET    | `/services`     | List/search/filter all 50 services        |
| GET    | `/departments`  | Department summary                        |
| POST   | `/feedback`     | 👍/👎 on a chat answer                     |
| GET    | `/analytics`    | Dashboard data (KPIs, top services, etc.)  |
| GET    | `/health`       | Backend status, active data store, and retrieval mode |

Interactive API docs are auto-generated by FastAPI at `/docs` (Swagger UI)
and `/redoc`.

## Known simplifications (by design, for a few-hour build)

- Retrieval uses Gemini-hosted embeddings + cosine similarity rather than a
  self-hosted vector database (e.g. ChromaDB with a local index) — a
  deliberate trade-off for deploy speed and reliability on a free hosting
  tier, not a missing feature. Startup takes ~10–20s longer while the 50
  services are embedded once.
- The 50 services are curated/synthetic sample data representing a
  realistic college's services, not scraped from a real institution.
